1. PART A — STALE-WHILE-REVALIDATE
================================================================================

The user's claim: "clicking refresh blocks the UI with a loading state
for the full ~30 seconds, even though the last successfully refreshed
data is already sitting in the database and could be shown instantly."

INVESTIGATION (round-12 pre-change state of client/src/pages/Dashboard.tsx)
  - Line 32-35:  `trpc.dashboard.get.useQuery(...)` — the standard
                  React Query v5 useQuery call (no custom staleTime,
                  no custom keepPreviousData; defaults apply).
  - Line 62:      `if (dash.isLoading) return <DashboardSkeleton />;`
                  — gates the skeleton on `isLoading` (initial load).
  - Line 145-152: `<Shell ... refreshing={refresh.isPending}>` —
                  passes the mutation's `isPending` to the Shell.
  - Line 38-39:   onSuccess calls `utils.dashboard.get.invalidate(...)`
                  which triggers a background refetch.

What React Query's default behavior is in this configuration:
  - isLoading = true  → only on the initial fetch (no cached data)
  - isFetching = true → whenever a fetch is in flight, INCLUDING
                        background refetches triggered by invalidate
  - data preserved    → React Query's default behavior keeps the
                        last-known data in `data` while a new fetch is
                        in flight (this is the "stale-while-
                        revalidate" pattern)

Therefore the EXISTING code is already showing old data during a
refetch. The user is wrong about the "blocks" claim — but the brief
is right that the user is missing the "your data is refreshing" signal.
The only visible indicator was the refresh button's spinner, which is
easy to miss.

WHAT THIS COMMIT ADDS

  1. client/src/pages/Dashboard.tsx (Shell component, new banner)

     A new <div role="status" aria-live="polite" data-testid="refresh-
     in-flight-banner"> that appears below the header when

  2. client/src/pages/Dashboard.swr.test.tsx (NEW, 6 tests)

2. PART B — PARALLELIZE fetchBaselines
================================================================================

The user's hypothesis: "Per the last live timing measurement,
fetchHierarchy (7.6s) -> main insights block (9.75s) -> fetchBaselines
(10.1s) currently run strictly sequentially."

DEPENDENCY AUDIT (inspected in server/meta.ts, refresh-bottleneck
root-cause §5, and confirmed by the round-12 grep)

  - fetchHierarchy    needs: nothing                       (no inputs)
  - insightsParallel  needs: nothing (10 calls, all read-levels)
  - fetchBaselines    needs: ONLY (token, accountId)         (no inputs)
  - accountTimezone   needs: ONLY (accountId)                (no inputs)
  - silencedAdRestore  needs: hierarchy (ad IDs) + insights
                            (dailyMaps / adPresence30d)        (gate)

  None of the four top-level phases depend on each other. The
  silencedAdRestore is the only gated phase, and it only needs to
  wait for {hierarchy, insights} — not for {baselines, timezone}
  which only feed the return object.

Restructure (server/meta.ts: doRefresh)

  Before (round-1..11):
    t0  fetchHierarchy (7.6s)
        insightsParallel (9.75s)
            silencedAdRestore (2.82s)
        fetchBaselines (10.1s)
        accountTimezone (0.4s)
    t_f  ≈ 30.67 s
    [async]
    return snapshot

  After (round-12):
    t0  ─┬─ fetchHierarchy (7.6s, 3 sequential paginated calls)
        ├─ insightsParallel (9.75s, 10 parallel)
        ├─ fetchBaselines (10.1s, 3 internal groups parallelized)
        └─ accountTimezone (0.4s)
    │
    ├── await {hierarchy, insights} ── → silencedAdRestore (2.82s)
    │
    └── (baselines, tz already done by t=10.1s; await cost = 0)
    │
    t_f  ≈ 10.1 s (max single phase) + 2.82 s (silencedAdRestore) = 13 s
    [async]
    return snapshot

  Net: 30.67 s → ~13 s projected on a real wearefforce-shaped
  account, a 57 % reduction. The internal fetchBaselines call was
  also parallelized: the 3 internal groups (ctrMedian90, cpmAvg14 +
  cpmNow via Promise.all, cpaMedian30) used to run serially inside
  fetchBaselines; now they run together (1× delay instead of 3×).

WHAT THE NEW fetchBaselines LOOKS LIKE

  const ctrP  = (async () => { /* 90d daily ctr */ })();
  const cpmP  = (async () => {
    const [j1, j2] = await Promise.all([
      graphGet("...last_14d cpm"),
      graphGet("...last_3d cpm"),
    ]);
    return { cpmAvg14, cpmNow };
  })();
  const cpaP  = (async () => { /* 30d daily cpa */ })();
  const [ctr, cpm, cpa] = await Promise.all([ctrP, cpmP, cpaP]);

This means fetchBaselines itself drops from ~3.5s of internal cost
to ~1.5s (one round-trip) on the mock simulator — the three
internal calls now overlap.

VERIFICATION (concurrency regression test)

  New test in server/meta.concurrent.test.ts: 'fetchBaselines runs
  concurrently with fetchHierarchy + insightsParallel (round-12,
  refresh-instant-feel Part B)'.

  Asserts:
    1. Wall-time < 4× CALL_DELAY_MS (parallel floor = 3×, serial
       regression = 5×)
    2. ≥ 10 of the 18 fetches share the same t=0 timestamp
       (proving all four top-level phases start together)

  Without the parallelization, this test fails on assertion 1
  (5× delay > 4× threshold). The test pins the parallel structure
  so a future "innocent" code-cleanup can't accidentally serialize
  the four top-level phases again.

3. WHAT THIS PR DOES NOT DO
================================================================================

  - Does NOT add a "Cancel refresh" button. The refresh is now
    <15s (was 30s); cancel UI is more complexity than it's worth.
  - Does NOT add optimistic updates (write user mutations to the
    cache before the server returns). The existing setCheck flow
    already does that for the checkbox-toggled-checks UX.
  - Does NOT parallelize fetchHierarchy itself. It's three sequential
    paginated calls (campaigns → adsets → ads) inside the function.
    Parallelizing those would require concurrent requests for the
    three lists, which is a different optimization (probably safe
    but out of scope for this PR).
  - Does NOT change the engine, the relevance filter, the verdict
    rules, or any client-side DecisionTable aggregation logic.
