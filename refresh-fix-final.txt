1. ROUND-7 CODERABBIT THREADS — STATUS
================================================================================

OPEN #1 (Minor, AsyncLocalStorage scope, server/meta.ts:500)
    Was: refreshMetrics.enterWith() persisted across awaits, leaking the
         finished refresh's counters into later work in the same process.
    Fix: refactored buildSnapshot + fetchAdDailyHistory to wrap each
         operation in refreshMetrics.run(store, async () => {...}) (round-8
         commit). doRefresh() inside buildSnapshot; doWork() inside
         fetchAdDailyHistory. The store is released the moment the callback
         returns, so other awaits (including the silenced-ad restore call)
         use their own scoped store, not a finished parent's.
    Status: RESOLVED.

OPEN #2 (Major, sparse-delivery divergence, server/meta.ts:721-735)
    Was: NEW daily7 === toDaily(rows7) ≠ OLD daily7 === last7(toDaily(rows30))
         for an ACTIVE ad with delivery in days 8-30 but NONE in the trailing
         7 calendar days (the "active-but-recently-silent" population). The
         old code would have fed the engine F1/F2/W2/S1 inputs from the
         trailing 7 of the 30d sparse rows. The post-fix code gave
         daily7=[] for that ad, the engine bailed on length<3/4, and those
         verdicts were silently lost. Self-review item 4 said: "an
         active-but-recently-quiet ad is often ITSELF a symptom of fatigue
         Meta has already acted on, and losing that signal hides the
         problem rather than fixing it."
    Fix: RESTORED the legacy semantics, scoped to that narrow population.
         - server/meta.ts:add filtering param to fetchLevelInsights
         - server/meta.ts:after the 10-call parallel block, detect silenced
           ads (adPresence30d.has(id) && last_7d.length < 3), then make ONE
           additional filtered `last_30d time_increment=1` call with
           `filtering: [{field: "ad.id", operator: "IN", value: [...]}]`.
           Cost bounded by silenced population (NOT the full 875-ad count).
         - server/meta.ts:for silenced ads, ad.daily7 = last7(toDaily(rows30))
           — the historical pre-fix pattern. All other ads keep the fast-
           path toDaily(rows7).
         - Tests (LOCKED in):
             server/daily7Slice.test.ts  "silenced ad w/ rows ONLY in
                 days 8-29" → asserts daily7 === last7(toDaily(rows30));
                 byte-equal to the OLD pre-fix pattern.
             server/engine.bottleneck.test.ts  "silenced-ad legacy
                 restore — F1 verdict fires on stale-but-real data" →
                 asserts the engine consumes the legacy daily7 and fires
                 F1/watch on an F1-pattern ad with delivery only on
                 days 8-14.
         - scripts/measure-refresh-timing.mock.ts:distinguishes the two
           cases (last_30d/1 + filtering param = expected silenced-ad
           restore; last_30d/1 with NO filtering = bottleneck regression).
           Regression-guard invariant preserved for the common-case
           silencedAdIds.length === 0 path.
    Status: RESOLVED.

Outside-diff (server/routers.ts:472-484)
    Was: c.daily30 ?? c.daily7 ?? [] short-circuit on empty daily30
         (post-fix: ad-level is []) → silently dropped every child ad's
         daily7 from the parent's aggregated series.
    Fix: applied mirror of the round-2 own-series fix:
         c.daily30 && c.daily30.length > 0 ? c.daily30 : c.daily7 ?? []
         (routers.ts: child iteration).
    Status: RESOLVED.

3. ARCHITECTURE — FINAL SHAPE
================================================================================

Three ad-level daily-call shapes coexist after round 8:

  (a) Hot path (verdict-critical, dense coverage)
      - level=ad, date_preset=last_7d, time_increment=1
      - Source for ad.daily7 for the active-recently population
      - Cheap: ~4.5s for 875 ads (7000 rows max)

  (b) Cheap 30d presence aggregate
      - level=ad, date_preset=last_30d, NO time_increment
      - Source for hadDelivery() (relevance filter)
      - ~1.2s for 875 ads (875 rows max — 1 per ad)

  (c) Laziness + selective legacy-restore (round 8)
      - level=ad, date_preset=last_30d, time_increment=1
      - Source for ad.daily7 ONLY for the silenced-ad population
        (filtered to those ad IDs only)
      - For accounts with silencedAdIds.length === 0: zero calls
      - For accounts with N silenced ads: ONE call, cost ∝ N rows

  (d) Lazy fetch on demand (already in PR)
      - Same shape as (c) but unbounded — full 875 ads × 30d
      - Triggered by user selecting 14d/30d/custom range
      - ~108s when it fires, off the refresh path

The presence aggregate (b) and the silenced-restore fetch (c) are NOT the
same call. (b) is one row per ad (presence); (c) is up to 30 rows per ad
(history). They share the date_preset parameter but not time_increment.

properties hold under the new contract, except for the silenced population

    (5-ad matrix).

    population receives the LEGACY last7(toDaily(rows30)) pattern, byte-
    equal to the pre-fix code's daily7 for that population. Verified in
    engine.bottleneck.test.ts (F1 fires against the legacy data) and
    daily7Slice.test.ts (silenced-ad daily7 === last7(toDaily(rows30))).

The verdict-legibility claim is now accurate. The earlier claim
"engine verdict fully stable for every ad" was overstated for the
