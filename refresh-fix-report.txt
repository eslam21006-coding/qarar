1. WHAT CHANGED
---------------
Server (`server/meta.ts`, `server/routers.ts`):

  A. The ad-level daily Meta call moved OFF the hot path's last_30d window.
     What used to be one slow call now becomes two faster ones:
       - Hot path  (verdict critical):  date_preset=last_7d, time_increment=1
                                         (≈4× less data than last_30d for 875 ads)
       - Cheap presence call          : date_preset=last_30d, NO time_increment
                                         (1 row per ad that delivered in 30d)

  B. The relevance filter (meta.ts hadDelivery, previously reading
     dailyMaps.get("ad").get(id).length > 0) now reads the cheap 30d
     presence aggregate at the ad level, which produces the EXACT same
     membership — and is ~30× cheaper (1 row per ad vs up to 30 rows per
     ad). Campaign / ad-set levels kept the historical dailyMaps check
     unchanged (their 34 + 186 objects already made last_30d cheap).

  C. Added `fetchAdDailyHistory(token, accountId, days)` — a new exported
     helper that does the heavy last_30d daily fetch on demand. Wired to a
     new tRPC procedure `dashboard.adDailyHistory` (server/routers.ts) that
     the client triggers ONLY when the user picks a wider range.

Client (`client/src/components/DecisionTable.tsx`):

  A. A `trpc.dashboard.adDailyHistory.useQuery` is enabled for range in
     {14d, 30d, custom} only. While the call is in flight, a tiny inline
     loader with crisp Arabic copy shows in the toolbar area:
       "جاري تحميل بيانات الـ {RANGE_LABEL}…"
     Once it resolves (or fails with a graceful fallback), the per-row
     daily range totals come from the lazy series instead of the empty
     cached `ad.daily30`.

  B. `aggregate()` accepts an optional `lazyDaily` parameter so the lazy
     series threads into the per-row totals without changing the
     historical code path.

Tests (`server/daily7Slice.test.ts`, `server/engine.bottleneck.test.ts`,
       `server/relevance.test.ts`):

  Three regression-proofs added, all GREEN:

  - `daily7Slice.test.ts` (5 tests) — proves ad.daily7 is byte-identical
    whether sourced from a direct `last_7d` fetch or from the trailing-7
    slice of a `last_30d` series. Three sub-tests: pure transformation,
    trailing-window semantics, endTo-end via mocked Meta (full 30-day ad
    produces a 7-element daily7 whose dates match Meta's last_7d preset;
    a quiet ad with rows ONLY outside the last 7d produces empty daily7).
    An additional test confirms the relevance filter's new adPresence30d
    path keeps an ad whose presence aggregate lists it but whose daily7 is
    empty.

  - `engine.bottleneck.test.ts` (4 tests) — runs the rule engine (runEngine)
    against a snapshot, then with the same `daily7` field but a 30-day
    `daily30` extended with 23 wild days, and asserts every verdict +
    reason + action string is byte-identical. Special-cases decayMap (K4)
    and fatigueSignals (F1): crafted fixtures trigger each rule; the
    extended twin must produce the same verdict. Plus a monotonicity
    sweep over every demo ad.

  - `relevance.test.ts` (3 tests) — proves the relevance filter's
    membership matrix is preserved: 30d-only delivery, recent-only
    delivery, today-only delivery, 3d-only delivery, never-delivered,
    ACTIVE-only. Membership matches.

  Updated (`server/meta.test.ts`): the old contract test that asserted
  3× last_30d daily calls per level was rewritten to assert the new
  shape (2× last_30d daily at campaign/adset, 1× last_7d daily at ad,
  1× cheap presence aggregate at ad).

7. KNOWN GAPS / NEXT STEPS (out of scope for this PR)
-----------------------------------------------------
  - The 'custom' range with > 30d is not supported — fetchAdDailyHistory
    returns 14 or 30 only. The DecisionTable will silently show 30 days
    worth of history for a custom request that goes back further. Add
    a per-range days parameter when that use case surfaces.
  - The lazy fetch is per-tab, not memoized across users with the same
    ad account. DecisionTable's React Query config (`staleTime: 60_000`)
    keeps the response in cache for 1 min; a server-side per-account LRU
    would be a future perf improvement but is not necessary right now.
  - The cheap presence aggregate call still issues 1 Graph request per
    refresh. If even that proves expensive on a future, larger account,
    we can fold it into the same response as the last_7d daily call via
    `fields=ad_id,spend` field selection (not modeled here — would
    change the response shape and break the byte-identity tests).
  - The simulation script (scripts/measure-refresh-timing.mock.ts) is a
    structural-verification tool only — real numbers must be re-measured
    against the live account on the next natural refresh.
