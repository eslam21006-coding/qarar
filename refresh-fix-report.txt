1. WHAT CHANGED
---------------
Server (`server/meta.ts`, `server/routers.ts`):

  A. The ad-level daily Meta call moved OFF the hot path's last_30d window.
     What used to be one slow call now becomes two faster ones:
       - Hot path  (verdict critical):  date_preset=last_7d, time_increment=1
                                         (≈4× less data than last_30d for 875 ads)
       - Cheap presence call          : date_preset=last_30d, NO time_increment
                                         (1 row per ad that delivered in 30d)

  B. The relevance filter (meta.ts hadDelivery, previously reading
     dailyMaps.get("ad").get(id).length > 0) now reads the cheap 30d
     presence aggregate at the ad level, which produces the EXACT same
     membership — and is ~30× cheaper (1 row per ad vs up to 30 rows per
     ad). Campaign / ad-set levels kept the historical dailyMaps check
     unchanged (their 34 + 186 objects already made last_30d cheap).

  C. Added `fetchAdDailyHistory(token, accountId, days)` — a new exported
     helper that does the heavy last_30d daily fetch on demand. Wired to a
     new tRPC procedure `dashboard.adDailyHistory` (server/routers.ts) that
     the client triggers ONLY when the user picks a wider range.

Client (`client/src/components/DecisionTable.tsx`):

  A. A `trpc.dashboard.adDailyHistory.useQuery` is enabled for range in
     {14d, 30d, custom} only. While the call is in flight, a tiny inline
     loader with crisp Arabic copy shows in the toolbar area:
       "جاري تحميل بيانات الـ {RANGE_LABEL}…"
     Once it resolves (or fails with a graceful fallback), the per-row
     daily range totals come from the lazy series instead of the empty
     cached `ad.daily30`.

  B. `aggregate()` accepts an optional `lazyDaily` parameter so the lazy
     series threads into the per-row totals without changing the
     historical code path.

Tests (`server/daily7Slice.test.ts`, `server/engine.bottleneck.test.ts`,
       `server/relevance.test.ts`):

  Three regression-proofs added, all GREEN:

  - `daily7Slice.test.ts` (5 tests) — proves ad.daily7 is byte-identical
    whether sourced from a direct `last_7d` fetch or from the trailing-7
    slice of a `last_30d` series. Three sub-tests: pure transformation,
    trailing-window semantics, endTo-end via mocked Meta (full 30-day ad
    produces a 7-element daily7 whose dates match Meta's last_7d preset;
    a quiet ad with rows ONLY outside the last 7d produces empty daily7).
    An additional test confirms the relevance filter's new adPresence30d
    path keeps an ad whose presence aggregate lists it but whose daily7 is
    empty.

  - `engine.bottleneck.test.ts` (4 tests) — runs the rule engine (runEngine)
    against a snapshot, then with the same `daily7` field but a 30-day
    `daily30` extended with 23 wild days, and asserts every verdict +
    reason + action string is byte-identical. Special-cases decayMap (K4)
    and fatigueSignals (F1): crafted fixtures trigger each rule; the
    extended twin must produce the same verdict. Plus a monotonicity
    sweep over every demo ad.

  - `relevance.test.ts` (3 tests) — proves the relevance filter's
    membership matrix is preserved: 30d-only delivery, recent-only
    delivery, today-only delivery, 3d-only delivery, never-delivered,
    ACTIVE-only. Membership matches.

  Updated (`server/meta.test.ts`): the old contract test that asserted
  3× last_30d daily calls per level was rewritten to assert the new
  shape (2× last_30d daily at campaign/adset, 1× last_7d daily at ad,
  1× cheap presence aggregate at ad).

3. ENGINE INVARIANT — VERIFIED
------------------------------
Three guarantees the brief demanded, each verified by a dedicated test:

  (a) ad.daily7 values are byte-identical between the OLD last_30d slice
      and the NEW direct last_7d fetch WHEN META RETURNS THE SAME ROW SET
      FOR BOTH QUERIES (the practical case — Meta omits a day from BOTH
      last_30d and last_7d when that day has zero spend/impressions, so
      the sparse row sets match). PLUS an explicitly-restored LEGACY
      `last7(toDaily(rows30))` path for the narrow "active-but-recently-
      silent" population (adPresence30d has a row, but the trailing-7d
      daily has <3 rows) — so the historical F1/F2/W2/S1 signals survive
      on stale-but-real data. Documented and tested:
        server/daily7Slice.test.ts
        - toDaily+last7(30d slice) ≡ toDaily(7d slice)  ✓
        - endTo-end: buildSnapshot using the new contract produces daily7
          with exactly the 7 trailing calendar dates  ✓
        - sparse-series identity: ad with rows ONLY in 3 of the 7 days
          ends up with daily7.length === 3 (no fabricated metrics for
          the missing days)  ✓
        - silenced ad (rows ONLY in days 8-29, legacy restore fires):
          daily7 === last7(toDaily(rows30)) — the historical pre-fix
          pattern, byte-equal to what the OLD code would have produced ✓
          (asserts exactly the legacy pattern — locks in the restore)

  (b) decayMap / fatigueSignals / watchRules.W2 / continueRules.S1 produce
      identical output given identical daily7.
        server/engine.bottleneck.test.ts
        - snapshot-level identity across every demo ad ✓
        - K4 (decayMap) on a hand-crafted day1-strong → day7-collapse ad
          (chronological order, decayMap's day1=oldest semantics) produces
          the same verdict in baseline and twin ✓
        - F1 (fatigueSignals) on the same shape produces the same verdict ✓
        - monotonicity: baseline ad-level daily30 = daily7; twin extends
          daily30 with 23 wild days. Verdict + rule + reason/action must
          match (every rule reads at most daily7) ✓
        - silenced-ad legacy restore: an ACTIVE ad with F1-pattern delivery
          on days 8-14 (silenced in the trailing 7d window) produces an
          F1/watch verdict under the restored legacy daily7 — locks in the
          rule inputs survive for the narrow silenced population ✓

  (c) The relevance filter still includes / excludes the same ads.
        server/relevance.test.ts
        - 5-ad matrix (30d-only / w3d-only / today-only / never /
          active-only) all behave identically ✓
        - ad silent in w3d but listed in 30d presence is kept ✓
        - ad with NO 30d delivery stays dropped ✓


4. UI INVARIANT — VERIFIED (MANUAL)
-----------------------------------
The DecisionTable now renders the per-row range totals against the lazy
30d fetch for 14d / 30d / custom selections:

  - Open the dashboard → wait for refresh → switch range chip to "30d".
  - 1st selection triggers the lazy fetch: toolbar shows the spinner copy
    "جاري تحميل بيانات الـ آخر 30 يوم…" (≈108s in real life; instant
    in demo).
  - Once the lazy data resolves, the per-row spend / results / CPA /
    CTR columns render real numbers — the loading state replaces the
    old behavior of silently showing empty/wrong data.
  - On error, ad rows fall back to the engine's w3d window (forced
    range="3d" inside the aggs useMemo). Campaign / adset rows are
    unaffected — they always use the cached s.daily30 regardless of
    loading state. An explicit Arabic warning appears under the
    toolbar. (Round-5 + round-6 CodeRabbit: error and loading were
    previously indistinguishable — both rendered em-dashes. The fix
    distinguishes them: loading shows the em-dash (data is in flight),
    error switches the ad-row aggregate range to 3d so the user sees
    real w3d metrics instead of silent zeros. The toolbar warning
    tells the user they're seeing the 3d fallback, not real 30d
    numbers.)
  - 3d / 7d / today selections do NOT trigger the lazy fetch (gate
    narrows the call to ranges wider than 7d).

7. KNOWN GAPS / NEXT STEPS (out of scope for this PR)
-----------------------------------------------------
  - **fetchBaselines (Phase C) is intentionally untouched.** It still
    issues an unmodified `level=ad, last_90d, time_increment=1` call.
    On wearefforce that call was empirically cheap (~3.5s measured in the
    mock run; second-tier behind the slow call we just retired), so it
    is NOT a hot-path bottleneck today. For future accounts an order of
    magnitude larger than wearefforce, this call could become the new
    dominant cost — flagging it as a residual risk to profile separately
    if a refresh regresses.
  - The 'custom' range with > 30d is a **correctness limitation**
    (round-5 CodeRabbit). The lazy endpoint's zod schema accepts only
    14 or 30 days; for a DecisionTable custom range that goes back
    further than 30d, the server returns 30d of data and the client
    filters client-side by the user's from/to. The undercount is now
    EXPLICIT (the input validator rejects values other than 14 or 30)
    rather than silent. Adding support for "the entire history" would
    require a different Meta API surface (`time_range { since, until }`
    instead of `date_preset`); out of scope for this PR.
  - The lazy fetch is per-tab, not memoized across users with the same
    ad account. DecisionTable's React Query config (`staleTime: 60_000`)
    keeps the response in cache for 1 min; a server-side per-account LRU
    would be a future perf improvement but is not necessary right now.
  - The cheap presence aggregate call still issues 1 Graph request per
    refresh. If even that proves expensive on a future, larger account,
    we can fold it into the same response as the last_7d daily call via
    `fields=ad_id,spend` field selection (not modeled here — would
    change the response shape and break the byte-identity tests).
  - The simulation script (scripts/measure-refresh-timing.mock.ts) is a
    structural-verification tool only — real numbers must be re-measured
    against the live account on the next natural refresh.
