1. SCHEMA GUARD — the load-bearing absence is now enforced by a test
================================================================================

  THE PROBLEM
    The T037 gate works by an ABSENCE: `uq_funnelSettings_user_account` must
    NOT be declared in drizzle/schema.ts. If it is, the next
    `drizzle-kit generate` emits it INTO drizzle/meta/_journal.json, and the
    next deploy applies it automatically and UNGATED — before the
    diagnose -> repair -> verify-clean cycle, against a table that may still
    hold duplicate rows. That is the exact production failure this entire
    investigation was about.

    Until now that absence was protected only by prose: a comment in
    schema.ts, a warning in tasks.md, a warning in data-model.md. Prose does
    not fail a build. A single well-intentioned edit — the very edit the task
    list used to INSTRUCT, before we fixed it — would have silently removed
    the gate.

  THE FIX
    server/t037Gate.test.ts, first describe block. It reads drizzle/schema.ts
    off disk and asserts the index is not declared in CODE. It runs as part of
    `pnpm test`, which CI already runs (.github/workflows/ci.yml), so no
    ci.yml change was needed — the check is a test, and the test IS the CI
    check.

  ONE SUBTLETY WORTH RECORDING
    A naive grep for the index name FAILS IMMEDIATELY — and correctly so.
    drizzle/schema.ts:94 deliberately NAMES the index inside the docblock that
    warns the reader not to declare it. A substring match cannot tell that
    warning apart from the declaration it warns against.

    Blindly "fixing" that failure by deleting the warning comment would have
    been the worst possible outcome: it would remove the one piece of prose
    explaining why the index is absent, leaving a future reader with an
    unexplained gap and no reason not to fill it.

    So the guard strips comments first (stripComments()) and asserts on the
    remaining code. It then ALSO asserts the warning comment still exists —
    because someone who deletes the explanation is one step away from
    re-adding the declaration.

  PROVEN TO ACTUALLY CATCH THE DEFECT
    A test that never fails is worthless, so I verified it by sabotage:
    injected `uniqueIndex("uq_funnelSettings_user_account")` as real code into
    drizzle/schema.ts and re-ran.

      × T037 gate — drizzle/schema.ts must not declare the unique index
        → drizzle/schema.ts declares "uq_funnelSettings_user_account" in code.
        Tests  1 failed | 7 passed

    schema.ts was then restored (git diff --stat: clean, no change committed).

  A SECOND ASSERTION, guarding the other direction
    The guard above would still pass if someone simply DELETED
    drizzle/0010_settings_unique_index.sql — the constraint would then exist
    nowhere and never be applied at all. So the test also asserts:
      - 0010 still contains the index, AND
      - drizzle/meta/_journal.json still does NOT reference 0010
    If 0010 ever enters the journal, the gate is dead and the test says so.

3. THE OPEN UNKNOWN — NOW ANSWERED
================================================================================

  From t037-bypass-check-independent.txt, Finding D, which I flagged as the
  central unknown because it could not be answered from inside the repo:

      "We cannot verify from the repository whether the production deploy runs
       `pnpm run db:push` at all."

  ANSWERED, confirmed directly from Manus:

      MANUS DEPLOYMENT DOES **NOT** RUN `pnpm run db:push` AUTOMATICALLY.
      MIGRATIONS ARE A FULLY SEPARATE, MANUAL STEP.

  This resolves possibility (b) of the two I laid out, and it changes what the
  verifier is actually for.

  WHAT THIS MEANS — and it cuts both ways:

    GOOD: A Manus deploy cannot apply an ungated migration, because a Manus
    deploy does not apply migrations at all. The nightmare scenario — a routine
    push silently adding the unique index to a table full of duplicates — is
    off the table. It was never reachable through the deploy path.

    LESS GOOD: The verifier is therefore NOT a deploy-time guard, despite its
    docstring saying so. It only runs when a human types `pnpm run db:push`. It
    is a guard on the OPERATOR, not on the pipeline. Its real value is that the
    person applying a migration by hand — the only person who can apply one at
    all — cannot do so against an unverified database without being told to
    stop.

    That is still worth having; it is arguably the ONLY thing worth having,
    given migrations are exclusively manual. But it means the runbook, not the
    pipeline, is what protects production. The gate is a seatbelt for the
    operator, not an airbag in the deploy.

  CI is the exception that still matters: .github/workflows/ci.yml DOES run
  `pnpm run db:push` against a throwaway mysql:8 service. That is where the
  verifier runs unattended — and where the fail-closed change is load-bearing,
  since a CI run without a database must not silently "pass" the gate.
