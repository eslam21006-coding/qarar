STEP 0 — ACQUIRE THE WORKFLOW-WIDE LOCK
================================================================================
Before any other step, acquire a MySQL named advisory lock that is
held for the entire STEPS 1 through 3 flow. This lock prevents two
operators from racing the `NOT EXISTS` check in STEP 2, and prevents
a concurrent migration from invalidating the backup between STEP 1
and STEP 3. The 0 timeout means "fail fast, do not queue".

The lock name is the SAME for the whole runbook (backfill AND
rollback), so a backup-then-rollback pair is automatically
serialised against any other run.

CONNECTION SCOPE — CRITICAL. The lock, the transaction in any
later block, and the COMMIT/RELEASE_LOCK at the end are all
CONNECTION-SCOPED. They do not survive a connection close. The
operator MUST run every block in STEPS 0 through 3.5 (and every
block in ROLLBACK, if invoked) on ONE persistent MySQL
connection — the same connection for the whole runbook. If the
operator is using `webdev_execute_sql` or a similar tool that
opens a new connection per call, the lock acquired in STEP 0
will be released as soon as the STEP 0 block returns, and
STEPS 1–3 will run with no protection. In that case the
operator MUST use a long-lived SQL execution path (a single
`mysql` CLI session, or one continuous connection from a
script) instead of running each block as a fresh call.

If the tool being used CANNOT guarantee one connection across
all blocks, the operator MUST treat the lock as advisory only
and rely on the MAINTENANCE WINDOW section to keep uncoordinated
ledger writers off the database.

================================ ROLLBACK BASELINE — WHICH BACKUP IS THE TRUTH ================================

The ROLLBACK section restores the live ledger from THE ORIGINAL
PRE-BACKFILL BACKUP, not from any backup created during recovery.
A backup taken AFTER a partial STEP 2 INSERT (e.g. during a
recovery flow where the original backup was deemed stale) would
already contain the partial backfill as a permanent part of its
contents; restoring from it would lock the partial state in,
not undo the partial backfill.

Concretely:
  - The ROLLBACK statement ALWAYS targets the original
    `__drizzle_migrations_bak_YYYYMMDD` table that was the
    FIRST backup created on the date of the runbook. The
    operator MUST identify this backup (the one whose (a)
    check was originally "non-empty" and whose validation
    passed) and use it for the ROLLBACK statement.
  - If recovery created a SECOND backup on a LATER date, that
    second backup is for INSPECTION and for any FUTURE
    independent run only. It is NOT a rollback target.
  - If the original backup is itself damaged or missing, the
    ROLLBACK CANNOT be performed safely; STOP and report to
    a human reviewer. There is no automatic fallback to a
    later-date backup.

The simplest way to keep these straight is to record the name
of the original backup at the time STEP 1's `SELECT COUNT(*)
AS backup_row_count` first returned a non-empty result. That
name is the only valid input to the ROLLBACK section.

WHY THIS ACTUALLY FIXES THE PROBLEM (technical note, for the record)

Verified directly against the installed drizzle source in this repo,
node_modules/drizzle-orm/mysql-core/dialect.js:

================================================================================
ROLLBACK (only if STEP 3 shows something wrong)
================================================================================
Not expected to be needed. Do not run this unless instructed after reporting
a bad STEP 3 result. It restores the bookkeeping table to its pre-backfill
contents from the STEP 1 backup, and again touches no application data.

ROLLBACK IS ATOMIC. The earlier wording said "no DELETE statements", but a
standalone DELETE before the restore INSERT is not safe — if the INSERT
fails (network drop, lock timeout, anything), the live ledger is left
empty, which is the worst possible state for `drizzle-kit migrate`. The
procedure below wraps both writes in a single transaction so either
both succeed or neither does, AND uses a MySQL named advisory lock
(`GET_LOCK`) for cross-session concurrency control.

`LOCK TABLES` is NOT used here because MySQL implicitly commits any
active transaction when `LOCK TABLES` is issued — that would defeat the
atomicity this procedure exists to provide. `GET_LOCK` does not have
that side effect; it is the right tool for cross-connection mutual
exclusion without disturbing the transaction state.

The procedure is a TWO-PHASE COMMIT-style flow: the destructive writes
happen inside the transaction, the operator READS the sanity-check
output, and only then issues a single explicit `COMMIT` or `ROLLBACK`
based on the observed counts. The transaction guarantees that the
destructive writes are invisible to other connections until COMMIT,
so a wrong outcome simply means running ROLLBACK — the live ledger is
restored to its pre-rollback state.

================================ PHASE 1 — DO THE WRITES ================================

--------------------------------- COPY BELOW ---------------------------------
-- 1. Cross-connection mutex. The lock name is the SAME as the one
--    acquired by the backfill flow in STEP 0 of the runbook
--    ('qarar.drizzle_migrations_backfill'). Sharing the name means
--    a backup-then-rollback pair is automatically serialised
--    against any other backfill-or-rollback pair, AND against any
--    new backfill that an operator might launch while rollback is
--    in progress.
--
--    There are two distinct run-time cases for this GET_LOCK:
--      (A) IMMEDIATE ROLLBACK: the operator discovered an error
--          between STEP 0 and STEP 3.5, in the SAME connection
--          that still holds the lock from STEP 0. In this case
--          the second GET_LOCK is a reentrant no-op (MySQL
--          returns 1 immediately because this session already
--          holds the lock) and the subsequent RELEASE_LOCK
--          below decrements the hold count — STEP 3.5 will be
--          the actual final release.
--      (B) LATER ROLLBACK: the operator already ran STEP 3.5
--          and the backfill connection is closed. In this case
--          the lock is no longer held; this GET_LOCK acquires it
--          from scratch and the subsequent RELEASE_LOCK releases
--          it for real.
--    In both cases the operator MUST proceed only when the
--    result is exactly 1.
--
--    GET_LOCK() can return three values:
--      1 — lock was acquired (or reentrantly held) by this connection
--      0 — lock is held by another connection (CONTINUE would race)
--      NULL — lock acquisition errored (server-side fault, killed
--             connection, etc.) — the lock state is unknown
--    The operator MUST proceed only when the result is exactly 1.
--    A result of 0 OR NULL means STOP and report; the ROLLBACK
--    must not run without a confirmed lock because the
--    destructive DELETE+INSERT could race with a live migration.
SELECT GET_LOCK('qarar.drizzle_migrations_backfill', 0) AS got_lock;
-- If got_lock <> 1 the operator MUST stop and report. Continuing
-- on got_lock = 0 could race with a live migration; continuing
-- on got_lock IS NULL is even worse because the lock state is
-- unknown. The procedure must not proceed past this point until
-- the lock is confirmed.

-- 2. Open the single transaction. Everything below until the
--    COMMIT in phase 2 is one all-or-nothing unit.
START TRANSACTION;

-- 3. Sanity check #1 — record the backup row count, AND record the
--    full (id, hash, created_at) tuple for each backup row. Both will
--    be cross-checked after the restore INSERT; a count match alone
--    would let a row whose `hash` or `created_at` was tampered with
--    slip through. The full-row snapshot is the basis for the
--    equality check in step 5b.
SELECT COUNT(*) AS bak_rows
  FROM `__drizzle_migrations_bak_YYYYMMDD`;

-- 4. The destructive writes. If either fails (constraint violation,
--    lock timeout, network drop) the operator issues ROLLBACK in
--    phase 2 and the live ledger is unchanged.
DELETE FROM `__drizzle_migrations`;
INSERT INTO `__drizzle_migrations`
  SELECT * FROM `__drizzle_migrations_bak_YYYYMMDD`;

-- 5a. Sanity check #2 — the live count must match the backup exactly.
--     The operator reads this BEFORE issuing COMMIT/ROLLBACK.
SELECT COUNT(*) AS live_rows
  FROM `__drizzle_migrations`;

-- 5b. Sanity check #3 — every row in the live ledger must have
--     IDENTICAL (id, hash, created_at) to the corresponding row in
--     the backup. A row-count match is necessary but not sufficient:
--     a tampered row could pass the count check while disagreeing on
--     the row contents. The operator must read this result and
--     confirm it is empty BEFORE issuing COMMIT. A non-empty result
--     means the restored ledger does not exactly match the backup;
--     issue ROLLBACK and report.
SELECT m.`id`, m.`hash`, m.`created_at`,
       b.`id`    AS bak_id,
       b.`hash`  AS bak_hash,
       b.`created_at` AS bak_created_at
  FROM `__drizzle_migrations` m
  LEFT JOIN `__drizzle_migrations_bak_YYYYMMDD` b
    ON b.`id` = m.`id`
 WHERE b.`id` IS NULL
    OR b.`hash` <> m.`hash`
    OR b.`created_at` <> m.`created_at`;
--------------------------------- COPY ABOVE ---------------------------------

================================ PHASE 2 — COMMIT OR ROLLBACK ================================

DO NOT paste phase 2 until you have read all three sanity-check results
from phase 1: `bak_rows`, `live_rows`, and the full-row comparison
in step 5b. ALL THREE must pass. Specifically:
  - bak_rows must be > 0 (the backup is not empty)
  - live_rows must equal bak_rows (no rows lost or gained)
  - the step-5b result must be empty (every row matches on
    id, hash, AND created_at — a count match alone is not enough)
If ANY of these fails, run the ROLLBACK block (not the COMMIT block).
If all three pass, run the COMMIT block.

TO COMMIT (the safe, correct outcome — all three checks pass):

--------------------------------- COPY BELOW ---------------------------------
COMMIT;
SELECT RELEASE_LOCK('qarar.drizzle_migrations_backfill') AS released;
--------------------------------- COPY ABOVE ---------------------------------

TO ROLLBACK (any check failed — DO NOT COMMIT):

SELECT RELEASE_LOCK('qarar.drizzle_migrations_backfill') AS released;
--------------------------------- COPY ABOVE ---------------------------------

A failed ROLLBACK means the ledger is in the same state it was before

(Keep `__drizzle_migrations_bak_YYYYMMDD` around for at least a week after a

================================================================================
MAINTENANCE WINDOW — REQUIRED AROUND THIS RUNBOOK
================================================================================
`pnpm run db:push` runs `drizzle-kit generate && drizzle-kit migrate`
WITHOUT acquiring the `qarar.drizzle_migrations_backfill` advisory
lock. The lock this runbook acquires (STEP 0 for backfill, the
GET_LOCK at the top of the ROLLBACK section) ONLY serialises
cooperating runbook sessions — it cannot block a separate `db:push`
invocation, a direct `drizzle-kit` call, or any other ledger-
writing path that does not know the lock name. A concurrent
ledger writer on an uncoordinated path can therefore mutate
`__drizzle_migrations` while the runbook is in progress.

For that reason, the entire runbook — backfill (STEPS 0 through
3.5) AND the ROLLBACK section, including the destructive
DELETE/INSERT that ROLLBACK performs — MUST run during a window
in which no uncoordinated ledger writer is active. The window
extends from the moment the FIRST runbook block (STEP 0 for
backfill, or the GET_LOCK at the top of ROLLBACK) acquires the
lock, until the moment the LAST runbook block (STEP 3.5 for
backfill, or the COMMIT-or-ROLLBACK block at the end of ROLLBACK)
releases it. During that window:

  - NO `pnpm run db:push` may be invoked.
  - NO `drizzle-kit generate` or `drizzle-kit migrate` may be
    invoked directly either — the same race exists.
  - NO production deploy that includes a migration may be
    triggered.

The window in practice is short (a few minutes for the operator
to paste each block and read the output), but it is a window. If
`db:push` is invoked during the window, STOP this runbook, wait
for `db:push` to finish, and start over from STEP 0 (the lock
will have been released by the now-closed `db:push` connection,
or by the live deploy connection).

(The deeper fix — wiring the same advisory lock into `db:push`
itself — is tracked separately. Until that is in place, the
maintenance window is the operator's protection.)
