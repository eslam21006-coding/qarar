READ THIS FIRST — ANSWER TO QUESTION 4 (the safety question)
--------------------------------------------------------------------------------
In plain language:

  NO. Nothing in this runbook touches your real data.

  Every statement below reads or writes exactly ONE table:
  `__drizzle_migrations`. That table is a bookkeeping notepad. It holds a
  list of "which migration files have already been run", and nothing else.
  It has three columns: an id number, a text hash, and a timestamp number.
  It contains no user data, no ad data, no settings.

  These four tables are NOT named anywhere in the statements below, are not
  read, are not written, and cannot be affected:
      adAccounts
      funnelSettings
      user
      audit_log

  The backfill flow itself (STEPS 1–3) is purely additive: one CREATE
  TABLE that makes a NEW backup table, and one INSERT that adds rows
  to the bookkeeping table only. No existing row is modified or
  removed by STEPS 1–3.

  The ROLLBACK section (only run if STEP 3 shows something wrong and
  an operator is instructed to use it) DOES include a DELETE, but
  that DELETE runs inside a single transaction with the restore
  INSERT, against the bookkeeping table only, and is released by
  either COMMIT or ROLLBACK. The live ledger is either fully
  replaced or fully untouched — never partially modified. A failed
  ROLLBACK leaves the ledger in its pre-rollback state.

  What this DOES change: it stops Drizzle from wrongly believing migrations
  0000-0009 were never run. Right now Drizzle thinks they are unapplied and
  would try to re-run them, which is what causes "table already exists" /
  "duplicate column" errors on deploy. After this backfill, Drizzle agrees
  with reality and skips them.

  Worst case if something goes wrong: the bookkeeping table has wrong rows
  in it. STEP 1 backs it up first, so it can be restored exactly. Your
  actual application data is not in the blast radius at all.

STEP 0 — ACQUIRE THE WORKFLOW-WIDE LOCK
================================================================================
Before any other step, acquire a MySQL named advisory lock that is
held for the entire STEPS 1 through 3 flow. This lock prevents two
operators from racing the `NOT EXISTS` check in STEP 2, and prevents
a concurrent migration from invalidating the backup between STEP 1
and STEP 3. The 0 timeout means "fail fast, do not queue".

The lock name is the SAME for the whole runbook (backfill AND
rollback), so a backup-then-rollback pair is automatically
serialised against any other run.

--------------------------------- COPY BELOW ---------------------------------
SELECT GET_LOCK('qarar.drizzle_migrations_backfill', 0) AS got_lock;
--------------------------------- COPY ABOVE ---------------------------------
  Expected: got_lock = 1 (the lock was acquired).

  If got_lock = 0, the lock is held by another session — STOP and
  report. Do not proceed; another operator or a live migration is
  using the same name. Wait for it to finish, then retry.

  If got_lock IS NULL, the lock acquisition itself errored (server
  error, killed connection, etc.) — STOP and report. Do not proceed;
  the lock state is unknown.

  Continue only when got_lock = 1. The lock is released either
  explicitly by STEP 3's RELEASE_LOCK or implicitly when this
  connection ends.

STEP 1 — BACK UP THE BOOKKEEPING TABLE

With the workflow-wide lock held, create the backup table. The lock
was acquired in STEP 0 and is released only in STEP 3.

The backup table is named with TODAY'S DATE (UTC) so a re-run today

Expected result: success. If it errors with "table already exists", the
backup was already taken earlier today. BEFORE re-using that backup,
the operator MUST run the four validation checks below IN ORDER. All

operator can paste and run it as-is.

  Check (a) — backup is non-empty. Empty means someone truncated it;
  a fresh backup is required, NOT a re-use of this one.

--------------------------------- COPY BELOW ---------------------------------

FROM `__drizzle_migrations_bak_YYYYMMDD`;
--------------------------------- COPY ABOVE ---------------------------------
  Expected: bak_row_count >= 1 (typically 1 — the original mystery row).

  Check (b) — every row in the backup is still in the live ledger,
  with the same `(id, hash, created_at)`. A row whose primary key
  matches but whose `hash` or `created_at` differs has been TAMPERED
  with between STEP 1 and now; the backup is no longer a faithful
  snapshot and the safe move is a fresh backup. A row missing
  entirely means the backup pre-dates a deletion — also stale.

--------------------------------- COPY BELOW ---------------------------------
SELECT b.`id`, b.`hash`, b.`created_at`,
       m.`id`    AS live_id,
       m.`hash`  AS live_hash,
       m.`created_at` AS live_created_at
FROM `__drizzle_migrations_bak_YYYYMMDD` b
LEFT JOIN `__drizzle_migrations` m
  ON m.`id` = b.`id`
WHERE m.`id` IS NULL
   OR m.`hash` <> b.`hash`
   OR m.`created_at` <> b.`created_at`;
--------------------------------- COPY ABOVE ---------------------------------
  Expected: zero rows. Any non-empty result means a backup row is
  missing from the live ledger, OR a backup row's hash or
  created_at disagrees with the live row at the same id; in
  either case the backup is STALE — take a fresh one.

  Check (c) — no row was added to the live ledger AFTER the backup
  was taken, AND no live row's hash or created_at was changed. This
  is the case a one-way "backup ⊆ live" check misses: a concurrent
  migration could have added or altered rows after STEP 1 ran, and
  re-using the now-stale backup would cause a later ROLLBACK to
  delete those newer rows or lose those changes. Any unexpected
  row here means the backup is STALE — take a fresh one.

================================ IF VALIDATION FAILED — RECOVERY ================================

When check (a), (b), or (c) fails, the existing backup is stale or
damaged. The operator MUST take a fresh backup, on a NEW date, and
re-run all four validation checks against the new name. The runbook
NEVER drops an existing backup — it creates a new one with a new
date and re-validates. The complete copy-paste sequence:

  1. Pick the new date as today's UTC date. Example: 20260722.
  2. Run the CREATE statement with the new date substituted for
     YYYYMMDD.
  3. Re-run checks (a) through (d) against the new backup name.
  4. If the new backup passes, proceed to STEP 2 using the new
     backup name in any reference.
  5. If the new backup also fails, STOP and report — the live
     ledger itself is in an unexpected state and a human must
     decide whether to proceed.

A stale backup is NEVER dropped as part of this procedure; the
operator or a human reviewer may drop it later, after the
post-success window, once the new backup has been confirmed
correct.

WHY THIS ACTUALLY FIXES THE PROBLEM (technical note, for the record)
================================================================================
Verified directly against the installed drizzle source in this repo,
node_modules/drizzle-orm/mysql-core/dialect.js:

    select id, hash, created_at from `__drizzle_migrations`
      order by created_at desc limit 1
    ...
    if (!lastDbMigration || Number(lastDbMigration.created_at) < migration.folderMillis) {
        <run the migration SQL>
        insert into `__drizzle_migrations` (`hash`,`created_at`) values(...)
    }

Two consequences:

 1. Drizzle reads ONLY the newest row and compares its `created_at` against
    each journal entry's `when`. Once the newest row is 1783972463571 (0009's
    `when`, from drizzle/meta/_journal.json), every migration 0000-0009 fails
    the `<` test and is skipped without running any SQL. That is the whole fix.

 2. The `hash` column is written but NEVER read back or compared. A hash
    mismatch cannot cause a re-run. The hashes below are supplied for ledger
    fidelity, not for correctness of the skip.

Table name/columns confirmed: default `__drizzle_migrations`
(id serial primary key, hash text not null, created_at bigint), in the
database named by DATABASE_URL. drizzle.config.ts sets no custom
migrationsTable or migrationsSchema, so the default applies.

drizzle/0010_settings_unique_index.sql is deliberately NOT in _journal.json
and therefore is NOT affected by this backfill. The real unique index remains
pending behind the separate T037 diagnose->repair->verify gate, exactly as
intended.


================================================================================
HASH PROVENANCE — how the values above were computed
================================================================================
Method (from node_modules/drizzle-orm/migrator.js, readMigrationFiles):
    hash = sha256_hex( entire raw bytes of drizzle/<tag>.sql )
    created_at = the `when` value of that tag's entry in
                 drizzle/meta/_journal.json
Computed from the actual committed files, not guessed.

LINE-ENDING CAVEAT — WHY THESE HASHES DIFFER FROM THE EARLIER DIAGNOSIS FILE
    This repo has core.autocrlf=true and no .gitattributes, so six of the ten
    .sql files are stored in git with LF but appear in the Windows working
    copy with CRLF. sha256 covers those bytes, so the two produce different
    hashes. The earlier table in db-migration-drift-diagnosis.txt used the
    Windows CRLF bytes. That was the wrong set for this purpose.

    `drizzle-kit migrate` will be run from the Manus/Linux checkout, where the
    files are the committed LF bytes. The table below (used in STEP 2) is the
    LF/committed set — the one that machine would compute. Verified with
    `git show HEAD:<file> | sha256sum` and cross-checked against
    `git ls-files --eol`.

    This distinction is cosmetic, not functional: per point 2 above, the hash
    is never compared at runtime.

================================================================================
ROLLBACK (only if STEP 3 shows something wrong)
================================================================================
Not expected to be needed. Do not run this unless instructed after reporting
a bad STEP 3 result. It restores the bookkeeping table to its pre-backfill
contents from the STEP 1 backup, and again touches no application data.

ROLLBACK IS ATOMIC. The earlier wording said "no DELETE statements", but a
standalone DELETE before the restore INSERT is not safe — if the INSERT
fails (network drop, lock timeout, anything), the live ledger is left
empty, which is the worst possible state for `drizzle-kit migrate`. The
procedure below wraps both writes in a single transaction so either
both succeed or neither does, AND uses a MySQL named advisory lock
(`GET_LOCK`) for cross-session concurrency control.

`LOCK TABLES` is NOT used here because MySQL implicitly commits any
active transaction when `LOCK TABLES` is issued — that would defeat the
atomicity this procedure exists to provide. `GET_LOCK` does not have
that side effect; it is the right tool for cross-connection mutual
exclusion without disturbing the transaction state.

The procedure is a TWO-PHASE COMMIT-style flow: the destructive writes
happen inside the transaction, the operator READS the sanity-check
output, and only then issues a single explicit `COMMIT` or `ROLLBACK`
based on the observed counts. The transaction guarantees that the
destructive writes are invisible to other connections until COMMIT,
so a wrong outcome simply means running ROLLBACK — the live ledger is
restored to its pre-rollback state.

================================ PHASE 1 — DO THE WRITES ================================

--------------------------------- COPY BELOW ---------------------------------
-- 1. Cross-connection mutex. Any concurrent migration run also tries
--    to acquire this lock; the 0 timeout means "fail fast, do not
--    queue". GET_LOCK() can return three values:
--      1 — lock was acquired by this connection
--      0 — lock is held by another connection (CONTINUE would race)
--      NULL — lock acquisition errored (server-side fault, killed
--             connection, etc.) — the lock state is unknown
--    The operator MUST proceed only when the result is exactly 1.
--    A result of 0 OR NULL means STOP and report; the ROLLBACK
--    must not run without a confirmed lock because the
--    destructive DELETE+INSERT could race with a live migration.
SELECT GET_LOCK('qarar.drizzle_migrations_rollback', 0) AS got_lock;
-- If got_lock <> 1 the operator MUST stop and report. Continuing
-- on got_lock = 0 could race with a live migration; continuing
-- on got_lock IS NULL is even worse because the lock state is
-- unknown. The procedure must not proceed past this point until
-- the lock is confirmed.

-- 2. Open the single transaction. Everything below until the
--    COMMIT in phase 2 is one all-or-nothing unit.
START TRANSACTION;

-- 3. Sanity check #1 — record the backup row count, AND record the
--    full (id, hash, created_at) tuple for each backup row. Both will
--    be cross-checked after the restore INSERT; a count match alone
--    would let a row whose `hash` or `created_at` was tampered with
--    slip through. The full-row snapshot is the basis for the
--    equality check in step 5b.
SELECT COUNT(*) AS bak_rows
  FROM `__drizzle_migrations_bak_YYYYMMDD`;

-- 4. The destructive writes. If either fails (constraint violation,
--    lock timeout, network drop) the operator issues ROLLBACK in
--    phase 2 and the live ledger is unchanged.
DELETE FROM `__drizzle_migrations`;
INSERT INTO `__drizzle_migrations`
  SELECT * FROM `__drizzle_migrations_bak_YYYYMMDD`;

-- 5a. Sanity check #2 — the live count must match the backup exactly.
--     The operator reads this BEFORE issuing COMMIT/ROLLBACK.
SELECT COUNT(*) AS live_rows
  FROM `__drizzle_migrations`;

-- 5b. Sanity check #3 — every row in the live ledger must have
--     IDENTICAL (id, hash, created_at) to the corresponding row in
--     the backup. A row-count match is necessary but not sufficient:
--     a tampered row could pass the count check while disagreeing on
--     the row contents. The operator must read this result and
--     confirm it is empty BEFORE issuing COMMIT. A non-empty result
--     means the restored ledger does not exactly match the backup;
--     issue ROLLBACK and report.
SELECT m.`id`, m.`hash`, m.`created_at`,
       b.`id`    AS bak_id,
       b.`hash`  AS bak_hash,
       b.`created_at` AS bak_created_at
  FROM `__drizzle_migrations` m
  LEFT JOIN `__drizzle_migrations_bak_YYYYMMDD` b
    ON b.`id` = m.`id`
 WHERE b.`id` IS NULL
    OR b.`hash` <> m.`hash`
    OR b.`created_at` <> m.`created_at`;
--------------------------------- COPY ABOVE ---------------------------------

================================ PHASE 2 — COMMIT OR ROLLBACK ================================

DO NOT paste phase 2 until you have read all three sanity-check results
from phase 1: `bak_rows`, `live_rows`, and the full-row comparison
in step 5b. ALL THREE must pass. Specifically:
  - bak_rows must be > 0 (the backup is not empty)
  - live_rows must equal bak_rows (no rows lost or gained)
  - the step-5b result must be empty (every row matches on
    id, hash, AND created_at — a count match alone is not enough)
If ANY of these fails, run the ROLLBACK block (not the COMMIT block).
If all three pass, run the COMMIT block.

TO COMMIT (the safe, correct outcome — all three checks pass):

--------------------------------- COPY BELOW ---------------------------------
COMMIT;
SELECT RELEASE_LOCK('qarar.drizzle_migrations_rollback') AS released;
--------------------------------- COPY ABOVE ---------------------------------

TO ROLLBACK (any check failed — DO NOT COMMIT):

--------------------------------- COPY BELOW ---------------------------------
ROLLBACK;
SELECT RELEASE_LOCK('qarar.drizzle_migrations_rollback') AS released;
--------------------------------- COPY ABOVE ---------------------------------

A failed ROLLBACK means the ledger is in the same state it was before
phase 1 — no rows were lost. The operator must report which check
failed (`bak_rows = 0`, `live_rows <> bak_rows`, or a non-empty
step-5b result) and stop; do not retry the rollback without guidance,
since a failed step-5b indicates tampering or partial restoration
that a human must investigate.

(Keep `__drizzle_migrations_bak_YYYYMMDD` around for at least a week after a
successful run, then it can be dropped. The backup table is NEVER
deleted as part of this procedure.)
