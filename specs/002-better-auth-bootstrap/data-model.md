# Data Model: Better Auth Bootstrap + Schema Reset (Phase A)

Phase A is **additive**. It introduces the four Better Auth tables and changes **nothing** about the existing schema. The deferred Phase B retype inventory is recorded at the end for traceability.

---

## New tables (added in Phase A — `drizzle/auth-schema.ts`)

These are generated by `@better-auth/cli` for the MySQL dialect from `server/auth.ts`. Column lists below reflect Better Auth's standard model plus the three additional `user` fields; exact generated names follow the CLI output.

### `user`

| id | string (varchar/text) PK | Better Auth identifier (string, not int). |
| name | string | |

| Field | Type | Notes |

**Validation / rules (Phase A):**
- `subscriptionStatus`, `ghlContactId`, `role` are never accepted from the sign-up payload (`input: false`).
- Admin create hook: `ADMIN_EMAIL` set **and** equal to `user.email` ⇒ `role="admin"`, `subscriptionStatus="active"`; otherwise no change.
- Email+password enabled; no email-verification step.

## Schema module wiring (`drizzle/schema.ts`)

- Add `export * from "./auth-schema";` (additive).
- **Unchanged**: `users` table, `User`/`InsertUser` types, and every existing table's columns/indexes.
- No name collision: new exports are `user`/`session`/`account`/`verification`; existing are `users`/`User` — distinct.

---

## Exported types (`server/auth.ts`)

- `BetterAuthUser = typeof auth.$Infer.Session.user` — includes `subscriptionStatus`, `ghlContactId`, `role`.
- `BetterAuthSession = typeof auth.$Infer.Session.session`.

## Unchanged this phase — Phase B retype inventory (DEFERRED, do NOT touch in Phase A)

The legacy `users` table stays. The following six tables keep their **integer** `userId` FK in Phase A. Phase B will retype each `userId` to `varchar(36)` (and drop `users`), atomically with the data-layer cutover:

| Table | Current FK | Phase B target | Source line |
|-------|-----------|----------------|-------------|
| `metaConnections` | `int("userId").notNull().unique()` | `varchar("userId", { length: 36 })` | schema.ts:46 |
| `adAccounts` | `int("userId").notNull()` | `varchar(36)` | schema.ts:66 |
| `funnelSettings` | `int("userId").notNull()` | `varchar(36)` | schema.ts:87 |
| `snapshots` | `int("userId").notNull()` | `varchar(36)` | schema.ts:127 |
| `actionChecks` | `int("userId").notNull()` | `varchar(36)` | schema.ts:144 |
| `verdictHistory` | `int("userId").notNull()` (in composite index) | `varchar(36)` | schema.ts:169 |

> Note: the original request named `metaConnections`, `adAccounts`, `insightSnapshots`, `funnelSettings`. The real table is **`snapshots`** (not `insightSnapshots`), and there are **two more** (`actionChecks`, `verdictHistory`). All six are captured here so the Phase B retype is complete. Each table's own `id` primary key (`int autoincrement`) stays integer — only `userId` is retyped.
